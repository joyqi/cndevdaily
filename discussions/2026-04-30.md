### Round 1 - 提名

**主持人**:
> 各位开发者朋友，欢迎来到今天的讨论会。我是 Joyqi。
> 
> 我们今天要从这 **49 篇候选文章**中挑出最值得深入阅读的几篇。  
> 目标不是“最多点击”，而是要找出那些**有工程价值、有思想深度、值得开发者投入时间分析的内容**。
> 
> 我建议大家从三个角度来选：
> 1. **长期技术价值** —— 能启发我们在架构或工程实践中重新思考的内容；
> 2. **现实影响力** —— 对当前开发生态、开源环境或安全趋势具有即时参考意义；
> 3. **深度与原创性** —— 能深入探索系统设计原理，而不是表层新闻。
> 
> 今天的讨论分两轮进行：  
> - **第一轮**：每位参与者提出自己认为最值得深入的 3 篇文章，并简要说明理由；  
> - **第二轮**：我们聚焦前几篇投票集中度最高的文章，做细读与争论。
> 
> 我先为大家开放第一轮发言。  
> 请每位参与者依次分享——  
> 你从这份列表中选出的 3 篇文章，以及选择它们的逻辑与期望讨论方向。  
> 
> 现在，第一轮发言正式开始。  
> 谁愿意先来？

**小前**:
> 我选这三篇：  
> 
> **#2 [Lobsters] Zed is 1.0**  
> Zed 的开发体验太有话题了。一个用 Rust 写的编辑器，号称比 VS Code 还快、UI 丝滑，终于正式版！我想看看他们在性能和插件系统上的取舍，毕竟前端一种新 IDE 出世，总是让人又激动又疲惫 😆  
> 
> **#23 [Lobsters] Warp is now open‑source**  
> Warp 开源这事非常值得关注。他们 UI 是用 Rust + Web 技术堆出来的，算是「terminal-as-a-product」路线的集大成。我想研究一下它的渲染层和命令输入交互的实现，对做前端的 CLI 工具超有参考价值。  
> 
> **#16 [Lobsters] The end of responsive images**  
> 这标题太炸。我得看看作者是不是在挑 responsive image 的毛病，还是提出了新方案。如果真有更优雅的做法能简化 `<picture>` 那套，我第一个举手支持——毕竟现在要搞各种 `srcset`、`sizes`，完全是开发者体验灾难。

**老架**:
> 我选以下三篇：  
> 
> **#5 FastCGI: 30 Years Old and Still the Better Protocol for Reverse Proxies**  
> 这篇从协议层面探讨老技术的生命力，挺符合我关注的“稳定性优先”。FastCGI的设计理念值得对比现代的HTTP/2和gRPC，看长期维护成本上的trade-off。  
> 
> **#17 How Linux 7.0 Broke PostgreSQL: The Preemption Regression Explained**  
> 系统内核调度改变导致数据库性能问题，这种案例对后端架构师很有启发。它能提醒我们：底层假设的变化，可能直接影响分布式系统的可靠性。  
> 
> **#33 DRAM Crunch: Lessons for System Design**  
> 内存瓶颈不是只靠纵向扩展能解决的事。文章如果能分析硬件限制如何影响系统架构设计，这对规划服务扩容路线和评估技术债很有价值。

**运维老王**:
> 我挑这三篇，都是实操价值高、和运维工作贴得紧的：  
> 
> **#6 / #7《Copy Fail: 732 Bytes to Root》**  
> 听起来像是一起通用 Linux 提权漏洞，732 字节就能拿 root，这类问题对生产环境风险极高。我要看的是漏洞机制、影响面和临时缓解手段，尤其看容器和最小镜像会不会中招，好做自动化检测。
> 
> **#17《How Linux 7.0 Broke PostgreSQL: The Preemption Regression Explained》**  
> 内核调度回归把数据库搞挂，这就是典型的“应用挂了都来怪我”的坑。得了解具体 preemption 改动在哪，后面 CI/CD 里得加 kernel 测试或者内核版本 gate。
> 
> **#24《GitHub is sinking》**  
> GitHub 现在一旦出问题，对 CI/CD 和 IaC 仓库影响面太大。想看看作者是危言耸听还是真有迹象，比如 Actions 不稳定、API限流、封号政策。这关系到要不要提前迁 Forgejo 之类。

## 第二轮讨论：内容评选

### 文章摘要

**How Linux 7.0 Broke PostgreSQL: The Preemption Regression Explained**
> 文章分析了 Linux 7.0 中调度器预emption机制的变化如何导致 PostgreSQL 在高并发负载下性能骤降。核心问题在于移除了传统的 **PREEMPT_NONE** 模式，使得新调度策略在高锁争用场景下频繁上下文切换，严重影响 PostgreSQL 的共享缓冲区访问效率。该研究揭示了操作系统调度细节对数据库系统性能的深层影响，并促使内核社区重新审视针对服务器工作负载的调度优化策略。

**Secure signatures without a private key**
> 这篇文章提出了一种利用 **ECDSA 公钥恢复技术** 的新方法，使得构建出的签名既**安全又可完全重现**，而无需任何人持有或暴露私钥。该方案解决了在 **可重现构建（reproducible builds）** 中涉及签名时无法复现的问题，使远程证明（remote attestation）过程可以将源代码与包含签名的构建产物精确对应，从而强化 **机密计算（confidential computing）** 的可信度与验证透明性。

**Warp is now open-source**
> Warp 宣布全面开源客户端代码，并引入以代理（agent）为核心的开发流程，由其云端代理编排平台 **Oz** 负责执行，OpenAI 为该开源仓库的创始赞助方。此举旨在推动一种“人类监督、代理执行”的新型软件协作模式，让社区共同参与定义与验证产品，从而加速高质量软件的构建。Warp 同时推出对多模型（如 Kimi、MiniMax、Qwen）的支持、更灵活的自定义体验及可编程配置文件，体现其在开放式智能开发环境（ADE）方向的技术创新与开放生态愿景。
